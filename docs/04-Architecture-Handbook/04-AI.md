# 04 — AI

La piattaforma ha due sottosistemi AI distinti: **Copilot clinico**
(LLM + tool) e **Analisi radiologica** (visione, ONNX/YOLO). Sono servizi
separati, integrati al core via tool e callback.

## 1. Copilot clinico

- **Motore**: Spring AI + OpenAI **GPT-4.1** (`app.ai.model=gpt-4.1`).
- **Function calling**: `DentalCareAiTools` espone strumenti al modello
  (es. ricerca/creazione/aggiornamento paziente, lettura agenda, dati clinici).
  Il modello invoca i tool, che girano dentro il contesto tenant autenticato →
  nessun accesso cross-tenant.
- **Chat**: `ChatController` + `ChatService`, cronologia in tabelle per-tenant
  (`chat_sessions`, `chat_messages`); risposte in **streaming SSE**.
- **Suggerimenti proattivi/contestuali**: `CopilotSuggestionService` +
  `CopilotController`, con contesto della schermata corrente.

### Prompt Manager
I prompt del copilot vivono in `dentalcare.ai_prompts` con override per-tenant,
editabili da **Impostazioni → AI**. Placeholder dinamici (es. `{{oggi}}`,
`{{ora}}`) risolti a runtime. Sorgenti default in risorse `ai-prompts/*.txt`.

### Ruoli e accesso
I tool clinici sono pensati per ruoli medici
(`dentist`/`hygienist`/`orthodontist`/`surgeon`). L'accesso agli strumenti è
mediato dal ruolo dell'utente autenticato.

## 2. Analisi radiologica (servizio Python)

- **Servizio**: `dentalcare-ai-service`, Python **FastAPI** + **ONNX Runtime**.
- **Modelli**: YOLO in formato ONNX —
  `dentex_fdi_v1.onnx` (numerazione denti FDI) e
  `dentex_disease_v1.onnx` (rilevazione patologie). File gitignored, distribuiti
  a parte sul server di deploy.
- **Input**: immagini radiologiche in **formato standard** (non DICOM nativo —
  vedi [05-DICOM](05-DICOM.md)).
- **Integrazione con il core**:
  - Il backend invia il job al servizio AI.
  - Il servizio risponde via **callback firmato HMAC**, verificato da
    `HmacVerifier` (secret condiviso `app.ai.hmac-secret` ↔ `AI_CALLBACK_SECRET`).
  - Lato backend: `service/ai/` — `AiInferenceClient` (chiamata),
    `PatientDocumentAnalysisService` (orchestrazione), `AnalysisReconciler`
    (riconciliazione risultati), `OdontogramSyncService` (sync con odontogramma),
    `SseEmitterRegistry` (notifica esito in tempo reale al frontend).

## 3. Automazione conversazionale (n8n)

Workflow n8n gestiscono prenotazioni/interazioni via WhatsApp/voce. Si
autenticano al backend con un **service-token** (rilasciato da
`POST /api/public/service-token` con header `X-N8N-Key`) e operano sugli endpoint
agenda/pazienti. Gli appuntamenti sono serializzati in offset `Europe/Rome`.

## 4. Storage degli asset AI

Documenti e immagini paziente su **MinIO** (S3); il servizio AI e il backend vi
accedono per input/output delle analisi.
